# 1. 스프링 JDBC와 JdbcTemplate

## 1.1 순수 JDBC와 복잡성

### 1.1.1 보일러플레이트 코드의 한계
- 데이터베이스 연동 과정에서 발생하는 커넥션 연결, SQL 실행 객체 생성, 결과 집합(ResultSet) 처리 등 매번 반복되는 기반 코드가 다수 존재함.

```java
public List<PostDto> findAll() {
    String sql = "SELECT id, title, author, created_at AS createdAt FROM post";

    Connection conn = null;
    PreparedStatement pstmt = null;
    ResultSet rs = null;

    List<PostDto> result = new ArrayList<>();
    try{ // 플랜 A
        // 1. 데이터베이스 연결(Connection 객체 생성)
        conn = DriverManager.getConnection(DB_URL, DB_USER, DB_PASSWORD);

        // 2. SQL 실행 객체 생성(Statement 객체 생성)
        pstmt = conn.prepareStatement(sql);

        // 3. SQL 실행
        rs = pstmt.executeQuery();

        // 4. 결과 처리(ResultSet 사용)
        while(rs.next()){
            PostDto post = PostDto.builder()
                .id(rs.getInt("id"))
                .title(rs.getString("title"))
                .author(rs.getString("author"))
                .createdAt(rs.getObject("createdAt", LocalDateTime.class)).build();

            result.add(post);
        }

    }catch(Exception e){ // 플랜 B
        System.out.println("에러 발생: " + e.getMessage());
    }finally{
        // 5. 생성된 리소스를 생성의 역순으로 해제
        try{ if(rs != null) rs.close(); } catch (Exception e){ }
        try{ if(pstmt != null) pstmt.close(); } catch (Exception e){ }
        try{ if(conn != null) conn.close(); } catch (Exception e){ }
    }

    return result;
}
```

- 스프링 프레임워크는 이처럼 복잡하고 반복적인 데이터베이스 리소스 반납 및 예외 변환 작업을 템플릿 콜백 패턴 기반으로 대행하여 개발자가 순수 SQL 쿼리 작성과 데이터 바인딩에만 집중할 수 있도록 지원함.
  - 템플릿 콜백 패턴: 고정된 실행 흐름(템플릿) 속에서 변경해야 하는 가변적 로직을 콜백(인터페이스/람다) 형태로 전달받아 유연하게 처리하는 디자인 패턴. 스프링 `JdbcTemplate`의 경우 커넥션 획득, 스테이트먼트 실행, 리소스 반납 등 고정된 작업 흐름을 템플릿이 정의하고, 개발자가 작성한 구체적인 SQL 실행 및 RowMapper 매핑 로직만을 콜백 함수로 넘겨받아 동적으로 조합하여 수행함.

### 1.1.2 예외 처리의 강제성
- java.sql 패키지의 거의 모든 메서드가 체크 예외인 SQLException을 던짐.
- 기술 종속적인 예외 처리가 서비스 계층까지 전파되어 상위 계층의 순수성이 훼손됨.

#### 1) 서비스 계층의 기술 종속성 발생
- 리포지토리 계층에서 발생하는 SQLException을 처리하지 않고 던질 경우, 상위 비즈니스 로직을 처리하는 서비스 클래스의 메서드 시그니처와 임포트문에 `java.sql.SQLException`과 같은 특정 데이터베이스 접근 기술용 API가 직접 드러나 결합하게 됨.
  ```java
  package net.likelion.bebc25.board.service;
  
  import java.sql.SQLException; // 특정 데이터베이스 기술용 체크 예외에 종속됨
  
  @Service
  public class PostService {
      private final PostRepository postRepository;
  
      // 비즈니스 로직을 처리하는 계층임에도 불구하고 throws 선언이 강제됨
      public void registerPost(PostDto post) throws SQLException {
          postRepository.save(post);
      }
  }
  ```

#### 2) 순수성 훼손으로 인한 한계
- 서비스 계층은 핵심 비즈니스 규칙과 흐름 제어에만 집중해야 하는 영역이나, 기술 종속적인 예외가 노출됨으로써 계층 간의 완전한 격리가 실패함.
- 향후 데이터 접근 기술을 JPA나 NoSQL 등으로 교체할 경우 데이터베이스 기술 관련 예외 클래스가 변경되므로, 무관한 서비스 계층의 코드 영역까지 연쇄적으로 열어 수정해야 하는 유지보수 상의 문제를 유발함.

#### 3) 스프링의 예외 처리 방식
- 데이터베이스 처리 과정에서 발생하는 SQL 기반 체크 예외(`SQLException`)를 가로채어 스프링 프레임워크가 정의한 최상위 비체크 예외인 `DataAccessException` 계층 구조로 자동 변환하여 던짐.
- 특정 벤더 데이터베이스나 접근 기술(JDBC, JPA 등)에 상관없이 일관된 예외 추상화 계층을 제공함에 따라 서비스 영역의 임포트나 시그니처에서 `throws` 절이 완벽하게 제거되어 상위 서비스 계층의 기술 종속성 오염 문제를 원천 해결함.

## 1.2 JdbcTemplate 개요 및 의존성 설정

### 1.2.1 JdbcTemplate 핵심 기능
- 커넥션의 획득 및 반납, 쿼리문 준비 및 실행, 예외 변환 등 저수준 JDBC API의 제어 흐름을 스프링 컨테이너가 대행함.
- 체크 예외인 SQLException을 스프링의 비체크 예외인 DataAccessException 계층 구조로 변환하여 예외 처리 편의성을 제공함.

### 1.2.2 의존성 설정 및 자동 구성
- build.gradle에 spring-boot-starter-jdbc 의존성을 추가하여 사용 가능함.
  ```groovy
  dependencies {
      implementation 'org.springframework.boot:spring-boot-starter-jdbc'
      runtimeOnly 'com.mysql:mysql-connector-j'
  }
  ```
- 스프링 부트는 application.properties 설정을 기반으로 DataSource와 JdbcTemplate 빈을 자동으로 생성 및 등록함.

## 1.3 RowMapper를 활용한 결과 매핑

- 데이터베이스 조회 결과인 `ResultSet`(표 형태의 데이터)의 각 행(Row)을 자바 객체(DTO 또는 Entity)로 어떻게 변환할지 규칙을 정의하는 매핑용 템플릿 인터페이스.
- `JdbcTemplate`은 데이터베이스 조회 결과를 받아올 때 내부적으로 반복문(while(rs.next()))을 돌면서, 개발자가 제공한 이 RowMapper 규칙을 매 행(Row)마다 호출시켜서 자바 객체 목록으로 조립해 반환함.

- RowMapper 인터페이스 구조
```java
@FunctionalInterface
public interface RowMapper<T> {
    // rs: 데이터베이스 조회 결과(ResultSet), rowNum: 현재 처리 중인 행의 번호(0부터 시작하는 인덱스)
    T mapRow(ResultSet rs, int rowNum) throws SQLException;
}
```

### 1.3.2 람다식 기반 매핑 예시
- RowMapper는 함수형 인터페이스이기 때문에 별도의 클래스를 만들거나 익명 클래스로 구현할 필요 없이 람다식으로 간결하게 매핑 로직을 작성할 수 있음
  ```java
  private final RowMapper<PostDto> postRowMapper = (ResultSet rs, int rowNum) -> {
      return new PostDto(
              rs.getInt("id"),
              rs.getString("title"),
              rs.getString("content"),
              rs.getString("author"),
              rs.getBoolean("secret"),
              rs.getObject("created_at", LocalDateTime.class)
      );
  };
  ```

## 1.4 JdbcTemplate 기반 CRUD 구현

### 1.4.1 데이터 등록, 수정, 삭제 (update)
- INSERT, UPDATE, DELETE 등 테이블 상태를 변경하는 작업에는 update 메서드를 사용하며, 반영된 행(Row) 수를 반환함.
  ```java
  public void save(PostDto post) {
      jdbcTemplate.update(
          "INSERT INTO post (title, content, author) VALUES (?, ?, ?)",
          post.getTitle(),
          post.getContent(),
          post.getAuthor()
      );
  }
  ```

### 1.4.2 단건 데이터 조회 (queryForObject)
- 기본 키 등을 통한 단건 조회 시 queryForObject 메서드를 활용하며, 조회 결과가 없거나 2건 이상인 경우 예외를 유발함.
  ```java
  public PostDto findById(int id) {
      return jdbcTemplate.queryForObject(
          "SELECT id, title, content, author, secret, created_at FROM post WHERE id = ?",
          postRowMapper,
          id
      );
  }
  ```

### 1.4.3 다건 데이터 조회 (query)
- 목록 조회 등 다건의 결과를 받아올 때 query 메서드를 활용하며, 결과가 없으면 빈 리스트를 반환함.
  ```java
  public List<PostDto> findAll() {
      return jdbcTemplate.query(
          "SELECT id, title, content, author, secret, created_at FROM post",
          postRowMapper
      );
  }
  ```

## 1.5 데이터베이스 접속 정보의 외부 격리 및 @Value 활용

### 1.5.1 외부 설정 파일 분리
- 데이터베이스 주소(URL), 사용자명, 패스워드 등 환경별로 달라지는 정보를 application.properties 파일에 키-값 쌍으로 정의함.(키는 자유롭게 정의 가능)
  ```properties
  db.url=jdbc:mysql://localhost:3306/board_db?serverTimezone=Asia/Seoul&characterEncoding=UTF-8&useSSL=false&allowPublicKeyRetrieval=true
  db.username=user1
  db.password=1111
  ```

### 1.5.2 @Value 어노테이션 기반 필드 주입
- 스프링 빈 클래스 내부의 필드에 @Value 어노테이션을 부착하여 설정 파일의 속성값을 직접 할당함.
  ```java
  @Component
  public class DbConfig {
      @Value("${db.url}")
      private String url;
      
      @Value("${db.username}")
      private String username;
  }
  ```

### 1.5.3 스프링 부트의 자동 구성 표준 데이터소스 설정
- 임의 변수명(예: `db.url`)에 값을 대입한 뒤 `@Value` 등으로 꺼내 수동 구성하는 방식과 달리, 스프링 부트가 규약해 둔 표준 키(Key) 이름으로 값을 기재하면 별도 Java 구현 코드 없이 자동으로 커넥션 풀(`DataSource`)을 자동 생성 및 빈으로 등록해 줌.
  ```properties
  # 드라이버 클래스 지정 (스프링 부트가 드라이버를 로드할 때 참고, 생략시 url 정보를 이용해서 자동으로 찾음)
  spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver

  # 데이터베이스 접속 주소
  spring.datasource.url=jdbc:mysql://localhost:3306/board_db?serverTimezone=Asia/Seoul&characterEncoding=UTF-8

  # 데이터베이스 접속 자격 증명 정보
  spring.datasource.username=user1
  spring.datasource.password=1234
  ```

## 1.6 데이터베이스 초기화 및 커넥션 풀 제어

### 1.6.1 초기화 스크립트 자동 실행
- 스프링 부트는 애플리케이션 시작 시점에 SQL 스크립트를 실행하여 데이터베이스 스키마와 기초 데이터를 자동으로 구축하는 기능을 지원함.
- 클래스패스 루트(src/main/resources) 아래에 schema.sql과 data.sql을 위치시키면, 스프링 부트가 구동되면서 테이블 생성 DDL과 초기 데이터 삽입 DML을 순차적으로 수행함.
- application.properties에 아래 설정을 적용하여 스크립트 실행 방식을 제어함.
  ```properties
  # SQL 스크립트 초기화 설정
  
  # always: 내장형/외장형 데이터베이스 여부와 상관없이 기동 시 무조건 초기화 스크립트(schema/data.sql) 실행
  spring.sql.init.mode=always

  # never: 초기화 스크립트 실행을 완전히 차단 (데이터 누적 및 보존을 위해 물리 DB 연동 시 필수 지정)
  # spring.sql.init.mode=never

  # embedded: 기본값으로 H2 등의 내장형 메모리 데이터베이스가 작동할 때만 초기화 스크립트 작동
  # spring.sql.init.mode=embedded
  ```

### 1.6.2 커넥션 풀과 HikariCP 관리
- 커넥션 풀은 데이터베이스와의 논리적 연결 통로(Connection)를 미리 생성하여 풀에 보관하고 재사용하는 기술로, 접속 시 발생하는 오버헤드를 낮추어 애플리케이션 반응 속도를 향상시킴.
- 스프링 부트는 고성능 JDBC 커넥션 풀 구현체인 HikariCP를 기본값으로 탑재하여 자동 설정함.
- 애플리케이션의 트래픽 규모에 맞게 application.properties에서 HikariCP 세부 설정을 최적화하여 관리함.
  ```properties
  spring.datasource.hikari.maximum-pool-size=10
  spring.datasource.hikari.minimum-idle=5
  spring.datasource.hikari.connection-timeout=30000
  spring.datasource.hikari.idle-timeout=600000
  ```

# 2. 스프링 Data JDBC

## 2.1 Spring Data JDBC 개요

### 2.1.1 핵심 철학 및 기술 비교
- Spring JDBC (JdbcTemplate): SQL 쿼리 중심의 데이터 접근 기술로, 반복적인 JDBC 커넥션 제어 흐름은 스프링이 대행하지만 SQL 작성 및 결과 매핑(RowMapper)을 개발자가 직접 코드로 제어함.
- Spring Data JDBC: 엔티티 객체 정보를 기반으로 단순 CRUD SQL을 프레임워크가 자동 생성하는 ORM(Object-Relational Mapping) 철학이 도입된 기술로, 개발자의 SQL 작성 및 데이터 매핑 부담을 최소화하여 생산성을 향상시킴.
- 동작 단순성: 하이버네이트 기반의 Spring Data JPA와 달리 지연 로딩(Lazy Loading), 영속성 콘텍스트(1차 캐시, 쓰기 지연), 프록시 객체 등 복잡한 내부 메커니즘을 배제하고, 데이터베이스 상태 변경 시 즉시 쿼리가 실행되어 결과를 직관적으로 반영함으로써 기술적 복잡도를 대폭 낮춤.

### 2.1.2 의존성 설정
- build.gradle에 spring-boot-starter-data-jdbc 의존성을 주입하여 라이브러리 추가.
  ```groovy
  dependencies {
      implementation 'org.springframework.boot:spring-boot-starter-data-jdbc'
  }
  ```
- spring-boot-starter-data-jdbc 의존성을 추가하면 내부적으로 spring-boot-starter-jdbc 의존성이 전이(Transitive) 관계로 자동 포함되어 라이브러리가 다운로드되므로, 프로젝트 내부에서 JdbcTemplate과 동시 혼용하여 구동 가능함.

## 2.2 엔티티 설계와 테이블 매핑

### 2.2.1 @Table 어노테이션
- 자바 클래스(객체)와 데이터베이스의 특정 테이블을 매핑하기 위해 클래스 선언부 상단에 부착하는 어노테이션.
- 어노테이션 내부에 구체적인 값을 명시하지 않으면 자바 클래스명(CamelCase)을 데이터베이스 표준 명칭(snake_case)으로 자동 매핑하여 참조 관계를 성립 (예: 클래스명이 PostDetail일 경우 post_detail 테이블과 연결)
- 연동할 테이블의 물리 명칭이 클래스 이름과 다를 때 테이블 이름을 명시적으로 선언 (예: `@Table("post")`).
  ```java
  // post_detail 테이블과 PostDetail 클래스를 일대일로 수동 연결하는 예시
  @Table
  public record Post(
      @Id int id,
      String title
  ) {}
  ```

### 2.2.2 @Id 어노테이션
- 데이터베이스 테이블의 기본 키(PK) 컬럼과 매핑할 자바 객체의 필드 변수 지정에 사용.
  ```java
  @Table
  public record Post(
      @Id int id, // PK
      String title
  ) {}
  ```

- 해당 필드가 null 값을 가지고 있을 경우, 스프링은 이를 아직 데이터베이스에 등록되지 않은 신규 데이터로 판정하여 데이터베이스 삽입(INSERT) 처리를 진행.
- 데이터베이스의 자동 증가(Auto-Increment) 메커니즘을 적용한 경우 테이블 저장 성공 시 데이터베이스가 할당한 PK 값을 자바 엔티티 객체의 식별자 필드에 자동으로 채워 넣어줌

### 2.2.3 @MappedCollection 어노테이션
- 부모-자식 관계의 테이블들을 연동할 때 자식 데이터를 담을 컬렉션(List 또는 Set 등) 필드에 선언하여 관계를 정의.
- idColumn 속성: 자식 엔티티 테이블에서 부모 엔티티 테이블을 참조하는 외래 키(FK, Foreign Key) 컬럼 명칭을 강제로 지정 (예: `idColumn = "post_id"`)
  ```java
  @Table("post")
  public record Post(
      @Id int id,
      String title,
      String content,
      @MappedCollection(idColumn = "post_id") Set<Comment> comments
  ) {}
  ```

- keyColumn 속성: 자식 엔티티가 자바 컬렉션의 `List` 구조일 때 순서를 나타내는 인덱스 번호를 저장하는 컬럼을 연결하거나, `Map` 구조일 때 맵의 키 값을 저장하는 컬럼 이름을 설정하는 데 사용.
  ```java
  @Table("post")
  public record Post(
      @Id int id,
      String title,
      String content,
      // List 컬렉션 사용 시 데이터베이스에서 순서 번호를 관리할 comment_order 컬럼을 keyColumn으로 연동
      @MappedCollection(idColumn = "post_id", keyColumn = "comment_order") List<Comment> comments
  ) {}
  ```
- Spring Data JDBC는 1대N 관계를 항상 단방향 연관 관계로 모델링하므로, 자식 엔티티의 변경 및 삭제 제어가 부모 엔티티의 저장 상태에 완전히 통합되어 작동.

## 2.3 CrudRepository 기본 CRUD 구현

### 2.3.1 CrudRepository 상속
- 스프링이 제공하는 CrudRepository 인터페이스를 상속하여 별도의 SQL이나 구현 클래스 작성 없이 데이터 접근 계층을 정의함.
  ```java
  public interface PostRepository extends CrudRepository<Post, Long> {}
  ```

### 2.3.2 서비스 계층의 역할 분리
- 비즈니스 정책 결정은 서비스 클래스가 처리하고, 데이터베이스 접근 및 변경은 리포지토리에 위임하여 논리 계층을 분할함.
  ```java
  @Service
  public class PostService {
      private final PostRepository postRepository;
      
      public PostService(PostRepository postRepository) {
          this.postRepository = postRepository;
      }
  }
  ```

## 2.4 쿼리 메서드와 @Query 어노테이션

### 2.4.1 쿼리 메서드 자동 생성 규칙
- 리포지토리 인터페이스에 특정한 명명 규칙을 적용하여 메서드 시그니처를 선언하면, 스프링이 해당 시그니처를 파싱하여 SQL 쿼리를 동적으로 생성하고 구동함.
- 주요 조회 쿼리 생성 메서드 접두사 규격:
  - find...By: 특정 조건에 부합하는 데이터를 조회할 때 선언하며, 단건(Entity) 또는 다건(List, Set) 결과를 반환.
  - exists...By: 특정 조건의 데이터 존재 여부를 검사하여 boolean 결과를 반환.
  - count...By: 조건에 일치하는 데이터 레코드 수를 세어 long 결과를 반환.
  - delete...By: 조건에 부합하는 로우를 삭제하고 영향받은 행의 수를 반환.

#### 1) 쿼리 메서드 조건 키워드 매핑 테이블
- 메서드 명칭 뒤에 추가하여 상세한 SQL WHERE 조건절을 형성하는 핵심 키워드 목록:

| 키워드 | 메서드 명칭 예시 | 실제 변환 SQL WHERE 조건절 |
| :--- | :--- | :--- |
| And | findByTitleAndContent | WHERE title = ? AND content = ? |
| Or | findByTitleOrContent | WHERE title = ? OR content = ? |
| Containing | findByTitleContaining | WHERE title LIKE '%?%' |
| Like | findByTitleLike | WHERE title LIKE ? |
| Between | findByCreatedDateBetween | WHERE created_date BETWEEN ? AND ? |
| LessThan | findByIdLessThan | WHERE id < ? |
| GreaterThan | findByIdGreaterThan | WHERE id > ? |
| In | findByStatusIn | WHERE status IN (?, ?, ...) |

#### 2) 정렬 및 페이징 객체 연동
- 대용량 데이터를 효율적으로 정렬 및 제한 조회하기 위해 `Pageable` 및 `Sort` 파라미터를 제공하여 LIMIT 및 ORDER BY 처리를 자동화함.
  ```java
  // 제목 키워드를 포함하는 데이터를 정렬 조건에 맞춰 페이징 조회하는 인터페이스 선언
  Page<Post> findByTitleContaining(String keyword, Pageable pageable);

  // 정렬 조건만 동적으로 할당하여 다건을 조회하는 인터페이스 선언
  List<Post> findByTitleContaining(String keyword, Sort sort);
  ```

### 2.4.2 @Query 어노테이션 기반 수동 정의
- 메서드 네이밍 규칙만으로 표현하기 어려운 복잡한 다중 조인(Join) 연산이나 DBMS 전용 최적화 쿼리가 필요한 경우 직접 작성한 SQL 문법을 결합하여 처리함.

#### 1) 파라미터 바인딩 제어
- SQL 구문 내부에 `:파라미터명` 지시자를 할당하고, 메서드 매개변수 상단에 `@Param("파라미터명")` 어노테이션을 부여하여 일치시켜 매핑함.
  ```java
  @Query("SELECT * FROM post WHERE title LIKE CONCAT('%', :keyword, '%')")
  List<Post> searchByKeyword(@Param("keyword") String keyword);
  ```

#### 2) 데이터 변경 쿼리 선언과 @Modifying 결합
- SELECT가 아닌 `UPDATE`, `DELETE`, `INSERT`와 같이 데이터베이스 테이블 구조의 상태 값을 변경하는 쿼리를 작성할 때 사용함.
- `@Query`와 함께 `@Modifying` 어노테이션을 의무적으로 부착해야 스프링 내부에서 실행 방식을 데이터 변경 트랜잭션용 업데이트 작업으로 인지하여 쿼리를 수행함.
- 해당 어노테이션을 누락한 채 데이터 변경 쿼리를 호출할 경우 비정상적인 접근 예외(InvalidDataAccessApiUsageException)를 유발
  ```java
  @Modifying
  @Query("UPDATE post SET content = :content WHERE id = :id")
  int updatePostContent(@Param("id") Long id, @Param("content") String content);
  ```

## 2.5 스프링 선언적 트랜잭션 연동

### 2.5.1 @Transactional을 통한 트랜잭션 경계 설정
- 트랜잭션 (Transaction) 은 복수의 데이터베이스 처리 작업을 하나의 논리적 단위로 묶어 무결성을 보존하며, 프록시 기술을 기반으로 동작함.
- 메서드 시작 시점에 트랜잭션을 실행하고, 예외 없이 반환되면 커밋을 수행하며 비체크 예외 발생 시 자동으로 롤백을 수행함.

### 2.5.2 예외 종류별 처리 정책
- 런타임 예외 (RuntimeException): 복구하기 불가능한 치명적 상태로 규정하여 시스템 차원에서 즉시 롤백을 단행함.
- 체크 예외 (Checked Exception): 비즈니스 흐름 제어를 위한 정보 전달용 예외로 판단하여 기본적으로 커밋을 수행하므로, 롤백을 위해서는 rollbackFor 속성을 정의해야 함.
  ```java
  @Transactional(rollbackFor = Exception.class)
  public void registerPost(Post post) {
      postRepository.save(post);
  }
  ```

### 2.5.3 트랜잭션 제어 속성
- 선언적 트랜잭션의 동작을 정밀하게 제어하기 위해 5가지 주요 속성을 활용함.

#### 1) 전파 속성 (Propagation)
- 기존에 진행 중인 트랜잭션이 있을 때 새로운 메서드가 호출되는 경우, 이를 어떻게 병합하거나 격리할지 정의함.
- REQUIRED: 기본값으로, 기존 트랜잭션이 존재하면 참여하고 없으면 신규 트랜잭션을 시작함.
- REQUIRES_NEW: 언제나 독자적인 신규 트랜잭션을 실행하여 호출한 측의 성공 여부와 무관하게 독단적으로 커밋 및 롤백이 가능함.

#### 2) 격리 수준 (Isolation)
- 여러 트랜잭션이 동시에 실행될 때 일관성을 유지하는 독립 수준을 설정함.
- DEFAULT: 데이터베이스의 기본 격리 수준을 추종함.
- READ_COMMITTED: 다른 트랜잭션에 의해 확정된 데이터만 읽을 수 있도록 통제함.

#### 3) 읽기 전용 속성 (readOnly)
- 데이터 변경이 발생하지 않는 단순 조회 메서드에 적용하여 메모리 리소스 낭비를 차단함.
- true로 활성화할 경우 하이버네이트 스냅샷 생성 등의 작업이 배제되어 시스템 성능이 개선됨.

#### 4) 롤백 대상 예외 정의 (rollbackFor)
- 기본적으로 비체크 예외에 대해서만 롤백을 수행하는 프레임워크의 규칙을 변경하여, 명시한 예외가 발생했을 때도 소급 롤백을 강제함.

#### 5) 타임아웃 (timeout)
- 정의한 실행 시간 안에 메서드 처리가 완료되지 못할 경우 강제로 트랜잭션을 중단하고 실패 처리하여 장애를 최소화함.



## 2.6 데이터 접근 예외 처리 및 트러블슈팅

### 2.6.1 스프링 예외 추상화와 예외 변환
- 특정 데이터베이스 기술이나 드라이버(Vendor) 종류에 종속되지 않는 일관된 예외 추상화 계층을 제공함.
- 데이터베이스 연동 과정에서 드라이버가 던지는 SQLException 체크 예외를 분석하여, 스프링의 DataAccessException 비체크 예외 계층 구조로 자동 변환함.
- 이를 통해 개발자가 특정 예외 클래스를 개별적으로 캐치하지 않고도, 무결성 제약조건 위반이나 SQL 구문 에러를 안전하게 식별하여 예외 복구를 구현할 수 있도록 유도함.

### 2.6.2 자주 발생하는 에러 상황 및 조치 방안
- BadSqlGrammarException: SQL 문법이 잘못되었거나 조회하려는 컬럼 및 테이블이 데이터베이스에 존재하지 않을 때 발생하며, 실행된 쿼리문의 오탈자와 스키마 매핑 상태를 검사하여 조치함.
- DuplicateKeyException: 테이블의 기본 키(PK)나 유니크(Unique) 제약 조건이 설정된 컬럼에 이미 존재하는 값을 중복 삽입하려고 할 때 발생하며, 삽입 전 조회 로직을 강화하거나 중복 검증 로직을 추가하여 조치함.
- DataAccessResourceFailureException: 데이터베이스 서버와의 접속이 끊어지거나 네트워크 단절로 인해 리소스를 확보할 수 없을 때 발생하며, 데이터베이스 접속용 URL 주소 및 포트 번호가 활성화 상태인지 점검함.
