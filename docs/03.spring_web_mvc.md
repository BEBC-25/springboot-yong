# 1. 스프링 부트 프로젝트 생성

## 1.1 IntelliJ 기반 프로젝트 생성

### 1.1.1 새 프로젝트 시작
- IntelliJ IDEA > File > New > Project...

### 1.1.2 Spring Boot 선택
- 생성창 좌측의 Generators 목록 중에서 Spring Boot 선택

### 1.1.3 프로젝트 기본 정보 입력
- Name: `spring-board`
- Location(프로젝트를 생성할 폴더 선택): `c/bebc25/springboot-yong`(예시)
- Language: Java
- Type: Gradle - Groovy
- Group: `net.likelion.bebc25`
- Artifact: `spring-board`
- Package name: `net.likelion.bebc25.board`
- JDK(컴파일러/런타임): 25 Eclipse Temurin
- Java(코드내의 Java 문법 버전): 25 
- Packaging: Jar
- Configuration: Properties
- Next 클릭

### 1.1.4 스프링 부트 버전 및 의존성 라이브러리 선택
- Spring Boot: 4.1.0
- Dependencies
  - Developer Tools > `Spring Boot DevTools`: 코드 변경 시 서버 자동 재시작 등의 개발 편의 기능 제공
  - Developer Tools > `Lombok`: 반복 적으로 작성하는 코드를 자동으로 생성하는 기능 제공
  - Web > `Spring Web`: 웹 애플리케이션 개발 지원 (Tomcat 서버 내장, MVC 기반의 웹개발 지원)
  - Template Engines > `Thymeleaf`: 서버 사이드 HTML 뷰 템플릿 동적 렌더링 지원
  - I/O > `Validation`: 데이터 입력 유효성 검증 지원
  
- Create 버튼 클릭

## 1.5 실시간 변경 감지 및 자동 재시작 설정
- Spring Boot DevTools 주요 기능
  - 스프링 부트 애플리케이션 개발 단계에서 자바 코드나 정적 리소스를 수정할 때, 매번 서버를 재시작하고 웹 브라우저를 새로고침해야 하는 번거로움을 줄여 개발 생산성을 높이는 도구
  - 자동 재시작 (Automatic Restart): 자바 소스코드(.java) 수정 시 클래스로더 교체를 통해 내장 톰캣 서버를 재시작하는 기능
  - 라이브 리로드 (LiveReload): 정적 자원이 갱신되었을 때 웹 브라우저에 실시간 리프레시 신호를 전송하여 화면을 자동으로 새로고침해 주는 기능
  - 템플릿/정적 자원 캐싱 비활성화: 타임리프 템플릿 파일이나 CSS, JS 파일이 수정되었을 때 화면에 즉시 로딩되도록 뷰 템플릿 엔진의 내부 리소스 캐시 설정을 기본적으로 무효화하는 기능

### 1.5.1 자동 재시작 설정
- IntelliJ IDEA는 소스 코드 수정 시 자동으로 컴파일을 빌드하는 기능이 꺼져 있으므로, 코드를 저장하더라도 DevTools가 변경을 감지하지 못함
- 따라서 코드가 갱신되는 즉시 백그라운드에서 프로젝트 컴파일을 강제 실행하고, 서버가 구동 중인 상태에서도 이를 즉각 반영할 수 있도록 아래 두 가지 설정을 적용
- IntelliJ IDEA > File > Settings...
  - Build, Execution, Deployment > Compiler 선택 후 우측 `Build project automatically` 옵션 체크 후 `Apply` 버튼 클릭
  - Advanced Settings 선택 후 스크롤을 내리면 Compiler 하위의 `Allow auto-make to start even if developed application is currently running` 옵션 체크 후 `OK` 클릭

### 1.5.2 라이브 리로드 설정
- 서버 설정
  - `src/main/resources/application.properties` 파일을 열고 아래 설정을 추가
    ```properties
    spring.devtools.livereload.enabled=true
    ```
    - 주의사항: Spring Boot 4.x 스펙부터는 보안 상의 이유로 라이브 리로드 서버의 활성화 기본값(Default)이 false로 변경되었으며, 해당 옵션이 Deprecated(향후 삭제 예정) 처리됨

- 웹 브라우저 설정
  - 원리: 스프링 부트 백엔드 내부에서 기동된 LiveReload 웹소켓 서버(포트 35729)와 크롬 브라우저의 확장 프로그램이 양방향 소켓 통신망을 형성함. 이후 컴파일이 감지되어 리소스가 변경되면, 서버가 소켓망을 통해 브라우저 측으로 새로고침 명령을 송신하고 확장 프로그램이 이를 수신하여 브라우저 화면을 강제로 갱신함
  - 설치: 크롬 웹스토어에서 [LiveReload Reborn](https://chromewebstore.google.com/detail/livereload-reborn/kdlcalkcmchabpgfhmjannmkbppggkck) 확장 프로그램을 추가
  - 활성화: 스프링 부트 서버 구동 후 웹 브라우저로 `http://localhost:8080`에 접속한 상태에서, 브라우저 우측 상단 툴바의 LiveReload Reborn 아이콘을 클릭하여 활성화 (아이콘의 우측 상단에 초록색의 동그라미가 생김)
  - 적용 확인: HTML 또는 CSS 파일을 수정하고 컴파일이 감지되면 서버를 직접 껐다 켜지 않아도 웹 브라우저가 알아서 실시간 새로고침되어 반영되는지 확인

# 2. HTTP 프로토콜
HTTP(HyperText Transfer Protocol)는 웹 브라우저와 웹 서버 간 텍스트 기반으로 데이터를 송수신하기 위한 프로토콜

## 2.1 주요 특징
- TCP 기반 프로토콜
  - 클라이언트와 서버가 연결을 수립한 후 메시지를 교환
  - 데이터를 패킷 단위로 전송하며, 수신 측에서 데이터를 검증한 뒤 응답을 보냄
  - 데이터 누락 시 재전송 요청을 처리하여 신뢰성이 높음
  - 주로 HTTP, FTP, SMTP 등 신뢰성이 중요한 통신에서 사용됨

### 2.1.1 장점
- 신뢰성: TCP를 기반으로 하여 데이터 전달의 신뢰성이 높음
- 범용성: 전 세계적으로 가장 널리 사용되는 웹 통신 프로토콜

### 2.1.2 참고
- HTTP/2와 HTTP/3는 기존 HTTP/1.x의 비효율성을 개선하기 위해 다중화, 헤더 압축, UDP 기반 전송 등의 기술을 도입

## 2.2 동작 방식
### 2.2.1 클라이언트의 요청 (Request)
- 클라이언트(주로 웹 브라우저)가 HTTP 요청 메시지를 서버로 전송
- 요청에는 필요한 자원(URL)이나 작업(GET, POST 등 요청 메서드 포함)이 포함됨

### 2.2.2 서버의 응답 (Response)
- 서버는 요청을 분석하여 필요한 작업을 수행
  + 파일 읽기, 데이터베이스 조회, 외부 시스템 연동 등
- 작업 결과를 바탕으로 응답 메시지를 생성하여 클라이언트로 전송

### 2.2.3 웹 브라우저의 처리
- 클라이언트는 서버 응답 데이터를 받아 파싱한 후 화면에 출력
- 서버의 응답을 받은 후 클라이언트와 서버 간 연결이 해제됨 (HTTP/1.x의 경우 기본적으로 비연결형(Connectionless) 프로토콜)

- `Accept: text/html,application/xhtml+xml,application/xml;q=0.9,image/avif,image/webp,image/apng,*/*;q=0.8`
  + 클라이언트가 선호하는 응답 콘텐츠 타입과 품질 지수(q값: 0~1, 생략시 1, 1이 가장 높은 선호도)
- `Accept-Encoding: gzip, deflate, br, zstd`
  + 클라이언트가 지원하는 압축 방식
- `Accept-Language: ko,en-US;q=0.9,en;q=0.8`
  + 클라이언트가 선호하는 언어와 우선순위
- `Cache-Control: no-cache`
  + 캐시 동작 제어
  + no-cache: 캐시된 응답을 사용하기 전 서버에 검증 요청 (항상 재검증, 304 Not Modified이면 캐시 사용, 변경되었으면 새 데이터 다운로드)
  + no-store: 응답을 캐시에 저장하지 않음
  + max-age=3600: 캐시 유효 시간(초 단위)
- `Connection: keep-alive`
  + 연결 유지 여부 (keep-alive: 연결 유지, close: 연결 종료)
- `Host: localhost:8080`
  + 요청을 보내는 서버의 도메인 주소와 포트
- `If-Modified-Since: Mon, 13 Jul 2026 09:13:15 GMT`
  + 캐시된 리소스의 Last-Modified 시간을 전송, 서버가 변경되지 않았으면 304 Not Modified 응답
- `If-None-Match: W/"191-19a1fcb18fe"`
  + 캐시된 리소스의 ETag 값을 전송, 서버가 변경되지 않았으면 304 Not Modified 응답
- `Referer: http://127.0.0.1:8080/index.html`
  + 요청이 시작된 이전 페이지의 URL (철자 주의: Referer는 실제로는 Referrer지만 명세상 오타로 Referer로 표기)
- `User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/141.0.0.0 Safari/537.36`
  + 클라이언트의 브라우저 및 OS 정보

### 2.4.1 상태 라인 (Status Line)
```
HTTP/1.1 200 OK
HTTP/1.1 201 Created
HTTP/1.1 404 Not Found
HTTP/1.1 500 Internal Server Error
```
- HTTP 버전(HTTP Version): HTTP/1.1 또는 HTTP/2 등
- 상태 코드(Status Code): 200(성공), 404(찾을 수 없음), 500(서버 에러) 등
- 이유 구문(Reason Phrase): OK, Created, Not Found 등 응답 상태를 간단한 텍스트로 표현

#### 주요 응답 상태 코드

##### 2xx (성공)
- `200 OK`: 요청이 성공적으로 처리됨, 응답 본문에 요청된 리소스가 포함됨
- `201 Created`: 요청이 성공했으며 새로운 리소스가 생성됨 (일반적으로 POST 요청)
- `204 No Content`: 요청은 성공했지만 응답 본문이 없음 (일반적으로 DELETE 요청)

##### 3xx (리디렉션)
- `301 Moved Permanently`: 요청한 리소스가 영구적으로 이동됨, 새 URL은 Location 헤더에 포함
- `302 Found`: 요청한 리소스가 임시로 이동됨
- `304 Not Modified`: 리소스가 변경되지 않아 캐시된 버전을 사용해도 됨

##### 4xx (클라이언트 오류)
- `400 Bad Request`: 잘못된 요청 구문으로 서버가 처리할 수 없음
- `401 Unauthorized`: 인증이 필요함, 로그인 정보가 없거나 만료됨
- `403 Forbidden`: 인증은 됐지만 권한이 없어 거부됨
- `404 Not Found`: 요청한 리소스를 찾을 수 없음
- `405 Method Not Allowed`: 요청한 HTTP 메서드가 허용되지 않음

##### 5xx (서버 오류)
- `500 Internal Server Error`: 서버 내부 오류, 예상치 못한 문제 발생

### 2.4.2 응답 헤더 (Response Headers)
```
Cache-Control: no-store
Last-Modified: Mon, 13 Jul 2026 01:43:37 GMT
Accept-Ranges: bytes
Content-Type: text/html
Date: Mon, 13 Jul 2026 01:56:52 GMT
Keep-Alive: timeout=60
Connection: keep-alive
Transfer-Encoding: chunked
```

- `Cache-Control: no-store`
  + 캐시 동작 제어 (no-store: 응답 결과를 브라우저 및 중간 캐시 서버에 임시 저장하지 않도록 제한)
- `Last-Modified: Mon, 13 Jul 2026 01:43:37 GMT`
  + 리소스가 마지막으로 수정된 시간 (If-Modified-Since와 함께 캐시 검증에 사용)
- `Accept-Ranges: bytes`
  + 서버가 `부분 요청(Partial Request)`을 지원함
  + 클라이언트가 `Range: bytes=1000-1999` 형식으로 바이트 단위의 범위 요청 가능
- `Content-Type: text/html`
  + 응답 바디의 미디어 타입
- `Date: Mon, 13 Jul 2026 01:56:52 GMT`
  + 응답이 생성된 시간
- `Keep-Alive: timeout=60`
  + keep-alive 연결의 최대 유지 시간(60초)
- `Connection: keep-alive`
  + 연결 유지 여부 (keep-alive: 연결 유지, HTTP/1.1에서는 기본값)
- `Transfer-Encoding: chunked`
  + 데이터 전송 인코딩 방식 (chunked: 청크 단위로 분할 전송)

### 2.4.3 응답 바디 (Response Body)
```html
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <title>스프링 게시판 - 목록</title>
  ...

  ...
</body>

```

- 성공 응답: 요청한 데이터를 응답 바디에 JSON, XML, HTML 등의 형식으로 반환
- 에러 응답: 에러 메시지와 함께 상태 정보 반환

## 2.5 HTTP의 특징

### 2.5.1 비연결성(Connectionless)
- 클라이언트가 요청을 보내고 서버가 응답하면 상호 연결을 해제함
- 서버는 클라이언트가 누구인지 신경 쓰지 않고 요청 정보만 분석해 적절한 응답을 만드는 데만 집중함
- 서버 구현이 단순해져 웹 발전에 큰 역할을 함

### 2.5.2 무상태(Stateless)
- 서버는 클라이언트의 요청 정보만 가지고 응답을 생성하며, 무상태 [^2] 특성으로 인해 클라이언트의 상태를 기억하지 않음
- 요청한 사용자가 누구인지, 이전에 어떤 작업을 요청했는지를 관리하지 않음
  + 웹이 발전하면서 클라이언트 정보를 저장할 필요가 생겼고, 이를 위해 Cookie, Session, JWT 등의 저장 방식이 도입됨

## 2.6 HTTP 주요 메서드
- HTTP 메서드는 클라이언트가 서버에 요청을 보낼 때, 대상 자원(Resource)에 대해 어떠한 행위(조회, 등록, 수정, 삭제)를 수행하고자 하는지 알려주는 기능
- 메서드의 성격을 규정하는 주요 특징 속성
  - 안전성 (Safe Methods): 요청을 여러 번 호출해도 대상 자원의 상태를 변경하지 않는 안전한 성질 (예: GET)
  - 멱등성 [^3] (Idempotent Methods): 동일한 요청을 한 번 실행한 것과 여러 번 연속해서 실행한 결과가 서버 리소스 상태 기준으로 항상 동일한 성질 (예: GET, PUT, DELETE)
  - 요청 바디 존재 여부: 전송할 데이터(Payload)를 URL에 얹느냐(GET) 혹은 HTTP 메시지 바디에 실어 전송하느냐(POST, PUT, PATCH)의 차이

### 2.6.1 GET
- 서버로부터 자원을 가져올 때 사용
  + 게시글 목록 조회
  + 게시글 상세 조회
  + 회원 정보 조회

### 2.6.2 POST
- 서버로 데이터를 보낼 때 사용
  + 게시글 등록
  + 회원 등록

### 2.6.3 PUT
- 서버 데이터 한 건의 전체 항목 수정 시 사용
  + 특정 게시글의 모든 항목 수정
  + 특정 회원의 회원 정보 전체 수정

### 2.6.4 PATCH
- 서버 데이터 한 건의 일부 항목 수정 시 사용
  + 특정 게시글의 제목 수정
  + 특정 회원의 비밀번호 수정

### 2.6.5 DELETE
- 서버의 데이터 한 건을 삭제할 때 사용
  + 게시글 삭제
  + 회원 삭제

# 3. 3티어 아키텍처와 MVC 패턴 설계

## 3.1 3티어 아키텍처의 이해

### 3.1.1 3티어 아키텍처의 개념
- 3티어 아키텍처는 전체 웹 시스템을 물리적 및 독립적인 세 개의 하드웨어 서버(혹은 네트워크 영역)로 분할하여 구축하고 운영하는 인프라 아키텍처 설계 패턴
- 각 티어는 독립적인 시스템 프로세스로 구동되며, 통신망을 통해 데이터를 주고받음으로써 특정 영역의 장애가 전체로 확산되는 것을 방지하고 독립적인 가동 환경을 형성

### 3.1.2 3티어 아키텍처의 물리적 구성 요소
#### 1) 프레젠테이션 계층 (Presentation Tier)
- 사용자 기기(컴퓨터, 스마트폰 등) 화면에 보여지는 영역
- 웹 브라우저(크롬, 사파리 등)가 이에 해당하며, 서버로부터 전달받은 파일(HTML, CSS, JavaScript)을 파싱해서 화면에 그리고 사용자의 입력을 서버로 전달하는 역할
- 사용자 기기의 성능을 직접 활용하며 서버와는 네트워크를 통해 연결

#### 2) 애플리케이션 계층 (Application Tier / Business Tier)
- 사용자가 요청한 회원가입이나 글쓰기 등 프로그램의 실제 핵심 기능을 실행하고 처리하는 영역
- 자바 코드로 작성한 프로그램이 가동되는 서버 환경으로, 스프링 부트 애플리케이션이 실행되는 곳이 이에 해당
- 입력된 데이터를 검증하고 상황에 맞게 계산 및 판단을 내리며, 데이터베이스에 필요한 데이터를 요청하거나 저장하도록 조율하는 역할

#### 3) 데이터 계층 (Data Tier)
- 프로그램이 다루는 모든 회원 정보나 게시글 데이터 등을 안전하게 저장하고 관리하는 데이터 창고 영역
- 데이터베이스 관리 소프트웨어(MySQL, Oracle 등)가 실행되는 전용 컴퓨터 환경이 이에 해당
- 애플리케이션 계층의 심부름(데이터 조회, 저장 등) 요청을 받아 실제 테이블에서 데이터를 꺼내거나 새로 집어넣는 역할

### 3.1.3 물리적 분리의 이점
- 자원 활용의 유연성: 특정 계층의 부하가 가중될 경우 해당 티어의 서버 장비만 추가로 증설하여 확장(Scale-out)하기 수월
- 유지보수 제어: 소프트웨어 구동 환경이 물리적으로 격리되어 있어 특정 서버의 운영체제나 소프트웨어 버전을 업그레이드해도 타 서버에 미치는 영향이 없음
- 인프라 보안성 제고: 핵심 자산인 데이터베이스 서버를 외부 망에 노출하지 않고 사설 네트워크에 은닉하며, 오직 애플리케이션 서버(WAS)를 통해서만 접근을 허용하여 인프라 취약성을 제어

## 3.2 Model 1과 Model 2 아키텍처
웹 애플리케이션 아키텍처는 제어 흐름과 뷰의 분리 수준에 따라 Model 1과 Model 2 구조로 구분

### 3.2.1 Model 1 아키텍처
- 특징: 하나의 웹 문서 파일(HTML 등) 내부에 화면을 그리는 디자인 코드와 데이터를 다루는 자바 프로그램 코드가 구분 없이 한곳에 섞여서 구동되는 고전적인 구조
- 장점: 구조가 복잡하지 않아 소규모 개발 시 빠른 동작 검증이 가능
- 단점: 단일 파일 안에 프로그램 연산 흐름과 화면 요소가 뒤섞여 있어 페이지가 커질수록 유지보수가 어려우며, 디자이너와 개발자 간의 역할 분담이 어려워 협업 생산성을 저하함

### 3.2.2 Model 2 아키텍처 (MVC 패턴)
- 특징: 클라이언트의 요청 처리(Controller), 비즈니스 로직(Model), 화면 출력(View) 역할을 명확히 격리한 구조
- 컨트롤러(Controller) 역할을 담당하는 서버의 자바 프로그램이 요청을 수신하여 연산을 실행하고, 그 결과를 Thymeleaf 같은 뷰 템플릿에 주입하여 화면을 동적으로 완성
- 장점: 프레젠테이션 영역과 비즈니스 영역의 결합도가 낮아 역할 분담 및 유지보수성 향상 가능. 스프링 MVC 프레임워크 동작 구조의 모태가 됨

## 3.3 MVC 패턴
- 사용자 인터페이스, 데이터 및 제어 로직을 상호 분리하여 개발하는 소프트웨어 디자인 패턴
- Model: 애플리케이션의 데이터 및 핵심 비즈니스 로직을 담당하는 영역
- View: Model의 데이터를 바탕으로 프리젠테이션 티어에 제공할 View(HTML, JSON 등)를 만드는 영역
- Controller: 사용자의 입력 요청을 수신하여 분석하고, 필요한 비즈니스 로직을 호출하여 Model의 상태를 변경한 뒤 결과를 View에 전달하는 조율 영역
- 식당의 운영 구조와 비교
  - Model은 주방의 요리사, 실질적인 음식 제조(비즈니스 로직 수행 및 데이터 처리)를 담당
  - View는 메뉴판 및 식탁의 음식 플레이팅, 손님이 직접 보고 소비하는 최종 시각 화면을 의미
  - Controller는 홀의 서빙 직원, 손님의 주문(요청 수신)을 주방(Model)에 전달하고, 조리된 음식을 손님 식탁(View)으로 서빙하는 중개 및 조율을 담당

## 3.4 계층별 클래스 설계 기법 (Layered Architecture)
웹 애플리케이션의 유지보수성 향상을 위해 각 클래스를 역할에 따라 자바 소스코드 수준에서 다음과 같이 세 개의 계층(Layer)으로 나누어 설계

### 3.4.1 컨트롤러 계층 (Controller Layer)
- 클라이언트의 웹 요청을 가장 먼저 수신하여 데이터 검증 등을 처리하고, 서비스 계층을 호출한 뒤 결과 화면을 반환하는 클래스 영역

### 3.4.2 서비스 계층 (Service Layer)
- 프로그램의 핵심 비즈니스 규칙을 구현하고 연산하는 영역이며, 여러 데이터베이스 연동 작업들의 안전한 실행 단위(트랜잭션)를 관리하는 클래스 영역

### 3.4.3 레포지토리 계층 (Repository Layer)
- 데이터베이스와 직접 연결되어 테이블에 데이터를 저장하고 가져오는 등 실제 물리적 데이터 접근을 전담하는 클래스 영역

### 3.4.4 클라이언트의 요청이 서버로 전달되는 과정
- 클라이언트 -> 애플리케이션 서버 (컨트롤러 -> 서비스 -> 레포지토리) -> 데이터베이스 서버

### 3.4.5 서버의 응답이 클라이언트로 전달되는 과정
- 데이터베이스 서버 -> 애플리케이션 서버 (레포지토리 -> 서비스 -> 컨트롤러 -> 뷰) -> 클라이언트

## 3.5 계층 간 데이터 전달 객체의 분류
컨트롤러, 서비스, 레포지토리 계층 간에 정보를 교환하거나 데이터베이스의 결과를 담아둘 때 활용하는 데이터 전달 목적용 전용 클래스

### 3.5.1 DTO (Data Transfer Object)
- 목적: 소프트웨어 계층 간에(예: Controller와 Service) 데이터를 전송하기 위한 데이터 전달용 객체
- 특징: 비즈니스 로직을 갖지 않고 오직 읽기/쓰기 필드(getter/setter 혹은 자바 record)만을 보유하는 단순 데이터 홀더 객체
  - record: 불변 데이터를 읽기 전용으로 전달하기 위해 Java 16부터 정식 도입된 클래스 형식으로, 필드 선언만으로 생성자와 조회용 메서드(필드명과 일치하는 getter), toString, equals 등을 컴파일러가 자동으로 생성해 주는 데이터 전용 클래스(setter는 생성되지 않음)
  ```java
  public record PostDto(int id, String title, String content, String author, LocalDateTime createdAt) {}
  ```

### 3.5.2 Entity
- 목적: 데이터베이스 테이블과 1대1로 직접 대응되어 매핑되는 영속성 객체
- 특징
  - 데이터베이스 내에서 고유하게 식별할 수 있는 키(ID)를 가지며, 테이블 레코드 하나의 데이터를 담고 있음
  - 주로 서비스 계층과 레포지토리 계층 내부에서 비즈니스 로직을 수행하고 데이터를 처리할 때 활용함
  - 보안 및 설계 격리를 위해 외부 컨트롤러 계층까지 직접 노출하지 않으며, 서비스에서 컨트롤러에 데이터를 반환할 때는 전용 DTO 객체로 변환해 전달하는 것이 표준 설계

### 3.5.3 DAO (Data Access Object)
- 목적: 데이터베이스와 직접 연결되어 데이터를 저장하고 조회하는 SQL 쿼리를 실행해 주는 데이터 접근 전용 객체

## 3.6 프로젝트 패키지 구조 설계
- 회원 기능(Member)과 게시글 기능(Post)처럼 두 개 이상의 업무 도메인이 공존할 때 설계하는 프로젝트 패키지 구조는 크게 두 가지 방식으로 분류함

### 3.6.1 계층 기반 패키지 구조 (Package by Layer)
- 기술적 역할(Controller, Service, Repository 등)을 최상위 패키지로 삼고, 그 하위에 모든 도메인 클래스들을 모으는 전통적인 방식
```
net.likelion.bebc25.board
├── controller
│   ├── PostController.java
│   └── MemberController.java
├── service
│   ├── PostService.java
│   └── MemberService.java
├── repository
│   ├── PostRepository.java
│   └── MemberRepository.java
├── entity
│   ├── Post.java
│   └── Member.java
└── dto
    ├── PostDto.java
    └── MemberJoinDto.java
```
- 특징: 프로젝트 규모가 작을 때 계층별 분류가 명확하여 구조 파악이 용이하나, 도메인이 늘어나면 패키지 내부에 많은 클래스가 만들어져서 작업 시 패키지를 크게 오가며 찾아야 하는 불편함이 발생

### 3.6.2 도메인 기반 패키지 구조 (Package by Domain)
- 업무 기능(post, member)을 최상위 패키지로 세분화하고, 각 도메인 패키지 내부에서 하위 기술 계층 패키지를 각각 격리하여 소유하는 방식
```
net.likelion.bebc25.board
├── post
│   ├── controller
│   │   └── PostController.java
│   ├── service
│   │   └── PostService.java
│   ├── repository
│   │   └── PostRepository.java
│   ├── entity
│   │   └── Post.java
│   └── dto
│       └── PostDto.java
└── member
    ├── controller
    │   └── MemberController.java
    ├── service
    │   └── MemberService.java
    ├── repository
    │   └── MemberRepository.java
    ├── entity
    │   └── Member.java
    └── dto
        └── MemberJoinDto.java
```
- 특징: 각 업무 도메인 단위의 강한 응집도를 보장하며, 향후 특정 기능을 마이크로서비스(MSA) 형태로 추출하고 격리하기 유리함

# 4. Spring MVC 컨트롤러 설계 및 웹 요청 핸들러 정의

## 4.1 Controller 어노테이션 정의 및 데이터 직접 응답
- HTTP 요청을 수신하여 비즈니스 로직을 연결하고, 그 결과를 클라이언트에게 직접 문자열이나 HTML 데이터로 반환하는 기초 웹 컨트롤러 설계 기법

### 4.1.1 컨트롤러 정의 및 응답용 어노테이션
- `@Controller`
  - 클래스 상단에 명시하여 해당 클래스를 스프링 컨테이너가 관리하는 웹 요청 핸들러(컨트롤러) 빈으로 등록함
- `@ResponseBody`
  - 메서드 상단 혹은 반환 타입 앞에 부여하여, 메서드가 리턴하는 값(HTML 텍스트, DTO 등)을 HTTP 응답 메시지의 바디(Body) 영역에 직접 기입하여 전송함
- `ResponseEntity<T>` 클래스
  - HTTP 응답의 3대 요소인 상태 코드(Status), 응답 헤더(Headers), 응답 바디(Body)를 자바 객체 형태로 정밀하게 제어하여 반환할 수 있게 스프링이 제공하는 전용 응답 객체
  - 일반 컨트롤러 메서드에서 DTO나 문자열을 단순히 리턴하면 응답 상태 코드가 무조건 `200 OK`로 나가지만, ResponseEntity를 반환 타입으로 사용하면 상황에 맞게 `201 Created`나 `400 Bad Request` 등 커스텀 상태 코드를 직접 지정해서 보낼 수 있음
    ```java
    // 사용법 예시: 응답 바디에 메시지를 실어 보내며, HTTP 상태 코드를 201 Created로 지정함
    @GetMapping("/api/ok")
    public ResponseEntity<String> successResponse() {
        return ResponseEntity
                .status(HttpStatus.CREATED)
                .body("데이터 저장 완료");
    }
    ```

### 4.1.2 웹 요청 경로 매핑용 어노테이션 및 주요 속성
- `@RequestMapping`
  - 클래스나 메서드에 부여하여 해당 컨트롤러가 처리할 공통 URL 경로 범위를 지정
  - 특히 클래스 레벨에 선언할 경우, 내부 개별 메서드에 정의된 매핑 경로들의 앞부분에 공통 접두사(Prefix) 형태로 결합되어 URL 중복 선언을 방지하고 통합 제어하기 수월함
  - 주요 속성
    - `value` 또는 `path`: 매핑할 URL 주소 경로를 지정
    - `method`: 처리할 HTTP 메서드 방식을 직접 지정 (생략할 경우 모든 메서드를 다 수용)
  
    ```java
    @RequestMapping(value = "/board/list.html", method = RequestMethod.GET)
    @ResponseBody
    public String getBoardList() {
        return "<html>...</html>";
    }
    ```

    ```java
    // 클래스 레벨에 공통 접두사 경로(/board) 선언
    @Controller
    @RequestMapping("/board")
    public class BoardController {

        // 최종 요청 주소: /board/list (GET 방식)
        @RequestMapping(value = "/list.html", method = RequestMethod.GET)
        @ResponseBody
        public String getBoardList() {
            return "<html>...</html>";
        }
    }
    ```
    
- `@GetMapping` 및 `@PostMapping`
  - 각각 HTTP GET, POST 요청을 수신할 세부 URL 매핑 경로를 단축형으로 지정할 때
  - 주요 속성
    - `value` 또는 `path`: 세부 URL 주소 경로를 지정
  
    ```java
    @GetMapping(value = "/list.html")
    public String getBoardList() {
        return "<html>...</html>";
    }
    ```

    ```java
    @PostMapping("/write")
    public String writePost() {
        // 게시글 등록 로직 구현
        // ...
        return "redirect:list.html"; // 지정된 경로(/board/list)로 브라우저가 다시 GET 요청을 전송하도록 리다이렉트 응답 처리
    }
    ```

    - 리다이렉트 (Redirect 방식)
      - 응답 헤더에 302 상태 코드와 Location 경로를 담아 전송하여 브라우저가 새 주소로 다시 GET 요청을 보내도록 유도함
      - 브라우저의 URL 주소창이 최종 목적지 경로로 변경되므로 새로고침 시 중복 처리(게시글 이중 등록 등) 문제를 제어하는 데 사용

## 4.2 HTTP 파라미터 매핑 방법
- 클라이언트가 전달한 요청 데이터를 컨트롤러 메서드의 매개변수로 전달 받는 방법
  - @RequestParam: HTTP 요청 파라미터(예: ?name=value)를 1대1 매핑하여 수집
    ```java
    // 요청 URL: /search?keyword=spring
    @GetMapping("/search")
    @ResponseBody
    public String search(@RequestParam("keyword") String keyword) {
        return "검색어: " + keyword; // 검색어: spring
    }
    ```

  - @PathVariable: URI 경로 자체에 포함된 변수 값(예: /posts/{id}에서 id)을 추출하여 바인딩
    ```java
    // 요청 URL: /posts/15
    @GetMapping("/posts/{id}")
    @ResponseBody
    public String getDetail(@PathVariable("id") int id) {
        return "조회된 번호: " + id; // 조회된 번호: 15
    }
    ```

  - @ModelAttribute: HTML 폼 데이터를 DTO 객체 필드에 일괄 바인딩(DTO의 기본 생성자 호출 후 setter를 이용해 값을 주입하므로 기본 생성자가 존재해야 함)
    ```java
    // HTML 폼에서 전송된 title, content, author 값을 PostDto 객체 필드에 자동 매핑함
    @PostMapping("/write")
    @ResponseBody
    public String writePost(@ModelAttribute PostDto post) {
        return "제목: " + post.getTitle();
    }
    ```

# 5. 뷰(View) 구현 및 렌더링

## 5.1 서버 사이드 동적 화면 생성 기술의 변천사

### 5.1.1 서블릿(Servlet) 기반의 화면 렌더링
- 자바 클래스 파일 내에 PrintWriter 객체를 이용해 HTML 마크업 문자열을 직접 하드코딩하여 응답 스트림으로 전송하던 방식
- 단순 오타 수정이나 레이아웃 변경 시에도 매번 자바 코드를 컴파일하고 재배포해야 하므로 개발 생산성 저하 극심

```java
@WebServlet("/users")
public class UserServlet extends HttpServlet {
    @Override
    protected void doGet(HttpServletRequest req, HttpServletResponse resp) throws IOException {
        resp.setContentType("text/html;charset=UTF-8");
        PrintWriter out = resp.getWriter();
        
        List<User> users = getUsers();
        
        out.println("<html>");
        out.println("<head><title>사용자 목록</title></head>");
        out.println("<body>");
        out.println("  <h1>사용자 목록</h1>");
        out.println("  <ul>");
        for (User user : users) {
            out.println("    <li>" + user.getName() + " (" + user.getEmail() + ")</li>");
        }
        out.println("  </ul>");
        out.println("</body>");
        out.println("</html>");
    }
}
```

### 5.1.2 JSP(JavaServer Pages) 기반의 화면 렌더링
- HTML 문서 구조 안에 JSP 지시문(Directive, `<%@ %>`), 스크립틀릿(Scriptlet, `<% %>`), 표현식(Expression, `<%= %>`) 등 전용 태그 문법을 활용해 페이지 설정 정보를 정의하고 자바 프로그래밍 코드를 작성하고 변수값을 출력하는 작업을 통해 서블릿 대비 렌더링 효율 향상
- JSP 파일은 런타임 시점에 WAS(Web Application Server)에 의해 자바 소스 파일(.java)로 자동 파싱된 후, 컴파일 과정을 거쳐 실제 서블릿 인스턴스로 변환 및 실행됨 (JSP는 서블릿 코드를 좀 더 HTML 친화적인 구조로 작성할 수 있도록 지원하는 대안)
- 화면을 표현하는 프레젠테이션 영역과 비즈니스 로직이 단일 파일에 어지럽게 섞여 스파게티 코드가 되기 쉬우며, 디자이너와의 협업 난이도 대폭 상승(모델 1 방식)

```jsp
<%@ page contentType="text/html;charset=UTF-8" language="java" %>
<%@ page import="java.util.List, net.likelion.User" %>
<%
    List<User> users = getUsers();
%>
<html>
<head>
    <title>사용자 목록</title>
</head>
<body>
    <h1>사용자 목록</h1>
    <ul>
        <% for (User user : users) { %>
            <li><%= user.getName() %> (<%= user.getEmail() %>)</li>
        <% } %>
    </ul>
</body>
</html>
```

### 5.1.3 MVC 패턴과 템플릿 엔진의 등장
- 비즈니스 로직은 컨트롤러와 모델에 위임하고, 화면 렌더링은 데이터 주입 기능만 갖춘 템플릿 엔진에 위임하도록 명확히 분리
- JSP가 지닌 단점(반드시 WAS가 구동 중인 상태에서만 화면 확인 가능)을 극복하고, 로컬 환경에서 단독 HTML 렌더링이 가능한 모던 템플릿 엔진 기술로 발전

- Thymeleaf 템플릿 엔진 코드 예시
```html
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<head>
    <meta charset="UTF-8">
    <title>사용자 목록</title>
</head>
<body>
    <h1>사용자 목록</h1>
    <ul>
        <!-- th:each와 th:text를 통해 동적 데이터를 매핑 및 출력 -->
        <!-- 태그 바디의 텍스트(하루 (haru@gmail.com))는 서버 없이 브라우저로 단독 실행 시 출력되는 샘플 데이터이며, 서버 구동 시 실제 데이터로 대체됨 -->
        <li th:each="user : ${users}" th:text="|${user.name} (${user.email})|">
            하루 (haru@gmail.com)
        </li>
    </ul>
</body>
</html>
```

## 5.2 스프링의 뷰(View)와 뷰 리졸버(ViewResolver) 아키텍처
- 스프링 웹 MVC는 화면 렌더링 책임을 컨트롤러로부터 완전히 분리하기 위해 뷰(View)와 뷰 리졸버(ViewResolver)라는 인터페이스 기반의 추상화 설계를 도입함
- 컨트롤러는 어떠한 화면 기술(JSP, Thymeleaf, Mustache 등)이 배후에서 돌아가는지 알 필요 없이 단순히 화면의 논리적 명칭만을 반환하도록 하여 결합도를 느슨하게 격리함

## 5.3 뷰 리졸버 (ViewResolver) 인터페이스
- 컨트롤러가 반환한 뷰의 논리적 식별 이름(예: "board/list")을 해석하여 실제 브라우저 응답 처리가 가능한 물리 뷰 객체(View)를 결정하고 찾아내는 책임을 담당하는 인터페이스
- 주요 특징 및 탐색 순위 체인:
  - 스프링 MVC는 여러 종류의 뷰 리졸버들을 빈으로 동시 등록할 수 있으며, 이들은 내부적인 우선순위(Order) 순서에 맞춰 등록 순서대로 체인을 이루어 매핑 파일을 스캔함
  - 주요 구현체
    - InternalResourceViewResolver: JSP 템플릿 파일 경로를 해석하여 서빙할 때 사용
    - ThymeleafViewResolver: 타임리프 템플릿 파일을 해석하여 서빙할 때 사용
    - FreeMarkerViewResolver, MustacheViewResolver: 각각의 전용 엔진 파일 경로를 해석하여 서빙할 때 사용

## 5.4 뷰 (View) 인터페이스
- 컨트롤러로부터 전달된 데이터(Model)와 클라이언트에게 전달할 최종 응답 자원(HTML 문서, JSON 문자열 등)을 실제로 그려내는(렌더링) 역할을 수행하는 인터페이스 (예: JSP 화면을 그리는 InternalResourceView, 타임리프 화면을 그리는 ThymeleafView 등)
- 핵심 메서드
  - `void render(Map<String, ?> model, HttpServletRequest request, HttpServletResponse response) throws Exception;`
  - 이 렌더링 메서드가 호출되면 각 View 구현체들은 모델 내부에 담긴 데이터를 읽어들여 템플릿 문서 상의 동적 위치에 바인딩 처리를 완료한 후 응답 스트림에 출력 데이터를 전달함

## 5.5 뷰 처리 렌더링 동작 시나리오
1. 컨트롤러가 비즈니스 연산을 완수한 후, 화면에 주입할 데이터(Model)와 이를 처리할 뷰 이름(예: "board/list")을 디스패처 서블릿(DispatcherServlet)에 반환함
2. 디스패처 서블릿은 등록된 뷰 리졸버(ViewResolver) 체인을 높은 순서(Order)부터 차례대로 호출하여, 전달받은 뷰 이름에 대응하는 뷰(View) 객체의 생성을 의뢰함
  - ThymeleafViewResolver의 경우 "templates/" + "board/list" + ".html" 경로를 조합하여 실제 템플릿 파일을 찾아내어 뷰 객체를 생성함
3. 뷰 리졸버가 특정 경로상의 물리 리소스를 매칭하여 뷰 객체(예: ThymeleafView 등)를 확보하여 디스패처 서블릿에 반환함
4. 디스패처 서블릿은 획득한 뷰 객체의 `render()` 메서드를 호출하며 매개변수로 모델 맵(Model Map)과 서블릿 응답 스트림 객체를 넘겨줌
5. 뷰 객체가 모델 데이터를 주입받아 완성한 최종 웹 리소스를 생성하고, 브라우저가 응답 스트림 객체를 이용해 출력을 수행함

# 6. Thymeleaf 뷰 템플릿 엔진
- 공식 홈페이지: https://www.thymeleaf.org

- 웹 애플리케이션의 동적 화면(View) 렌더링을 위해 사용하는 자바 진영의 서버 측 템플릿 엔진
- 주요 특징(내추럴 템플릿): 기존 HTML 표준 태그 구조를 훼손하지 않고 HTML 태그에 속성 방식으로 추가해서 작동하여, 서버 구동 없이 웹 브라우저에 html 파일을 단독으로 더블클릭하여 열어도 정적 화면 레이아웃이 깨지지 않고 정상 표시됨
- 스프링 통합 장점: Spring EL 표현식(`${}`) 지원, 스프링 폼 데이터 바인딩 및 다국어 메시지 맵, 스프링 시큐리티 등 스프링 MVC 인프라와의 완벽한 연동 기술을 제공함

## 6.1 컨트롤러 데이터 바인딩 및 템플릿 파일 경로 규칙
### 6.1.1 컨트롤러와 Model 객체
- 컨트롤러는 클라이언트 요청을 수신하면 스프링으로부터 Model 객체를 전달받음
  - 컨트롤러 메서드의 매개변수로 지정
    ```java
    public String getBoardList(Model model){
        // 게시글 목록 조회(데이터)
        List<PostDto> posts = getPosts();
        log.debug(posts.toString());

        model.addAttribute("posts", posts);
        return "board/list"; // 뷰 템플릿의 이름
    }
    ```
- Model 객체는 뷰 템플릿과 바인딩시켜서 최종 뷰를 만들기 위한 데이터 역할을 함
- model.addAttribute(key, value): 뷰가 동적으로 렌더링할 때 꺼내어 쓸 수 있도록 지정한 key 명칭으로 실제 자바 객체(value)를 모델에 보관함. 이후 뷰 템플릿에서 key 값으로 사용 가능

### 6.1.2 뷰 템플릿 파일
- 컨트롤러로부터 전달받은 데이터(Model)를 HTML 마크업 코드와 동적으로 결합하여 최종적인 웹 화면을 생성해 내는 템플릿 파일
- 역할: 정적 HTML 문서 양식에 `th:text`와 같은 타임리프 전용 속성들을 부여해 두면, 타임리프 렌더링 엔진에 의해 최종적으로 순수 HTML 표준 마크업으로 자동 변환 처리되어 브라우저에 전송됨
- 저장 위치: 스프링 부트 웹 프로젝트에서 타임리프 템플릿 파일의 표준 기본 저장 경로는 `src/main/resources/templates/` 디렉터리
- 파일 매핑 및 흐름: 컨트롤러에서 논리적 뷰 명칭으로 `board/list`를 문자열로 최종 반환하면, 스프링 MVC의 뷰 리졸버가 결합되어 실제 물리 파일 경로인 `src/main/resources/templates/board/list.html` 파일을 찾아내 동적 데이터 바인딩 렌더링을 시작함

## 6.2 타임리프 기본 문법
- 타임리프 속성을 사용하려면 HTML 파일 최상단 html 태그에 네임스페이스(`xmlns:th="http://www.thymeleaf.org"`)를 명시해야 함
  ```html
  <html lang="ko" xmlns:th="http://www.thymeleaf.org">
    ...
  </html>
  ```

### 6.2.1 변수값 출력
- `th:text`: 컨트롤러에서 Model 객체에 담아 전달한 변수값을 타임리프 템플릿에서 꺼내어 태그 내부의 텍스트로 안전하게 주입함
- 변수 표현식 `${...}`을 사용해 Model에 저장된 key 이름으로 객체를 참조함

- 컨트롤러 코드 예시
  ```java
  @GetMapping("/detail.html")
  public String getDetail(@RequestParam("id") int id, Model model){
      PostDto post = getPost(id);

      model.addAttribute("post", post);
      return "board/detail"; // src/main/resources/templates/board/detail.html
  }
  ```

- 타임리프 템플릿 코드 예시
  ```html
  <td th:text="${post.title}">세 번째 게시글 제목 샘플</td>
  ```

- 렌더링 결과
  ```html
  <td>1번 게시글</td>
  ```

### 6.2.2 조건문 사용하기
- 지정한 조건식의 참 또는 거짓 여부에 따라 특정 HTML 요소를 화면에 선택적으로 노출
- `th:if`: 조건식 평가 결과가 참일 때 해당 HTML 요소 렌더링
- `th:unless`: 조건식 평가 결과가 거짓일 때 해당 HTML 요소 렌더링
- 단순 텍스트 변경은 삼항 연산자가 유리하지만, 노출할 태그 종류나 UI 레이아웃 구조 자체가 아예 달라질 때는 th:if(참일 때 노출) 및 th:unless(거짓일 때 노출) 속성 조합이 편리함
  ```html
  <!-- 비밀글인 경우: 자물쇠 아이콘과 함께 단순 텍스트로 안내 메시지 출력 -->
  <span th:if="${post.isSecret}">
    <img src="/icons/lock.png" alt="lock"> 비밀글입니다.
  </span>

  <!-- 공개글인 경우: 상세 보기 링크가 활성화된 하이퍼링크 태그 출력 -->
  <a th:unless="${post.isSecret}" th:href="@{detail.html(id=${post.id})}" th:text="${post.title}">게시글 제목 샘플</a>
  ```

  - 비밀글일 때 최종 렌더링되어 출력되는 HTML 결과
    ```html
    <span>
      <img src="/icons/lock.png" alt="lock"> 비밀글입니다.
    </span>
    ```

  - 공개글일 때 최종 렌더링되어 출력되는 HTML 결과(post.id가 2이고 post.title이 "2번 게시글"인 경우)
    ```html
    <a href="detail.html?id=2">2번 게시글</a>
    ```

### 6.2.3 반복문 사용하기
- `th:each`: 리스트나 배열 등 컬렉션 데이터를 순회하며 동일한 HTML 마크업 요소를 반복하여 렌더링함
- 반복 변수와 변수 표현식(`${}`)을 조합하여 루프를 구현함
  ```html
  <tr th:each="post : ${posts}">
    <td th:text="${post.id}">0</td>
    <td th:text="${post.title}">게시글 제목</td>
  </tr>
  ```

  ```html
  <tr>
    <td>1</td>
    <td>1번 게시글</td>
  </tr>
  <tr>
    <td>2</td>
    <td>2번 게시글</td>
  </tr>
  ```

### 6.2.4 링크 사용하기
- 서버 배포 설정에 따른 컨텍스트 경로(application.properties 파일에 server.servlet.context-path로 지정된 서브 경로)를 런타임에 자동으로 감지하고 결합하여 정적/동적 URL 링크를 안전하게 조립함
- 하드코딩된 상대 또는 절대 경로 사용 시 배포 경로 변경에 따라 링크가 깨지는 문제를 방지하며, 배포 독립성과 이식성을 제공함
- 링크 표현식 `@{...}`과 함께 사용하며, 괄호 `(...)`를 이용해 경로 변수를 주입하거나 쿼리 파라미터를 추가 지정할 수 있음

- 경로 변수(Path Variable) 주입 예시
  ```html
  <a th:href="@{posts/{id}/edit(id=${post.id})}">수정하기</a>
  ```

- 경로 변수 예시의 최종 렌더링 결과(post.id가 2인 경우)
  ```html
  <a href="posts/2/edit">수정하기</a>
  ```

- 쿼리 파라미터(Query Parameter) 추가 예시
  ```html
  <a th:href="@{posts(page=${page}, searchKeyword=${searchKeyword})}">목록</a>
  ```

- 쿼리 파라미터 예시의 최종 렌더링 결과(page가 3이고 searchKeyword가 java인 경우)
  ```html
  <a href="posts?page=3&searchKeyword=java">목록</a>
  ```

### 6.2.5 유틸리티 객체 사용하기
- 날짜 형식 변환, 문자열 처리, 숫자 포맷팅 등 템플릿 내부에서 자주 수행하는 헬퍼 연산을 처리해 주는 타임리프 내장 유틸리티 객체를 제공
  ```html
  <!-- 날짜 및 시간 포맷팅 (#temporals) -->
  <span th:text="${#temporals.format(post.createdAt, 'yyyy-MM-dd HH:mm:ss')}">2026-07-13 18:00:00</span>

  <!-- 숫자의 천 단위 콤마 포맷팅 (#numbers) -->
  <span th:text="${#numbers.formatInteger(post.views, 3, 'COMMA')}">1,250</span>
  ```

### 6.2.6 폼 데이터 바인딩
- 서버에서 전달받은 객체(DTO)의 필드와 웹 브라우저 화면의 입력 창(input)을 연결하여 값을 서로 주고받을 수 있게 연동해 주는 기능
- 입력 창에 데이터를 채워 서버로 전송하면 지정된 DTO 객체의 필드에 값이 자동으로 담기며, 반대로 DTO 객체에 담겨 있던 기본값을 화면 입력 창에 자동으로 띄워 보여줄 수도 있음
- 주요 속성 및 바인딩 문법
  - `th:object`: 폼 데이터의 기준이 되는 DTO 객체(Model에 담긴 객체)를 바인딩함
  - `th:field`: DTO 객체의 개별 필드를 input 입력 창과 1대1로 연결하며, 브라우저가 화면을 띄울 때 필요한 HTML 기본 속성(id, name, value)들을 백그라운드에서 자동으로 생성하고 채워 줌
  - 선택 변수 식 `*{필드명}`: `th:object`로 지정한 객체의 내부 필드를 참조할 때 변수 표현식(`${모델명.필드명}`) 대신 간단히 단축 참조할 수 있는 문법

- @ModelAttribute 자동 모델 등록
  - 컨트롤러 메서드 파라미터에 @ModelAttribute("모델명")을 지정하면 스프링 MVC가 DTO 인스턴스를 자동으로 생성하고 요청 값을 바인딩하여 Model에 자동 주입하므로 개발자가 직접 model.addAttribute()를 호출할 필요가 없음
  - 등록 시 이름을 생략하면 클래스명의 첫 글자를 소문자로 바꾼 이름이 사용되며, 등록 화면(GET) 요청 시에도 빈 DTO 객체를 생성하여 자동으로 Model에 주입하므로 코드 작성을 간결화할 수 있음

  - 예시
    ```java
    // 게시글 등록 화면을 요청하는 컨트롤러
    @GetMapping("/write.html")
    public String getWriteForm(@ModelAttribute("postForm") PostDto post){
        return "board/write";
    }
    ```

    ```html
    <form th:action="@{write}" th:object="${postForm}" method="post">
      <input type="text" th:field="*{title}" placeholder="제목을 입력하세요">
    </form>
    ```

  - 실제 브라우저 렌더링 결과물 (th:action, th:object, th:field가 표준 속성으로 변환 처리됨)
    ```html
    <form action="write" method="post">
      <input type="text" id="title" name="title" value="" placeholder="제목을 입력하세요">
    </form>
    ```

### 6.2.7 리터럴 대체 문법
- 변수 표현식과 일반 정적 문자열을 문자 결합 연산자(+) 없이 편리하게 결합하여 사용할 수 있도록 제공하는 편의 문법
- 전체 결합 문자열을 파이프(|) 기호로 감싸서 표현하며, 가독성 향상 및 싱글 쿼테이션(') 사용의 번거로움이 없어짐
  ```html
  <!-- 파이프 기호를 사용하지 않은 방식 (더하기 연산자와 싱글 쿼테이션 수동 결합) -->
  <span th:text="${user.name} + ' (' + ${user.email} + ')'">샘플 텍스트</span>

  <!-- 파이프 기호를 사용한 방식 (간결하고 직관적인 결합) -->
  <span th:text="|${user.name} (${user.email})|">샘플 텍스트</span>
  ```

### 6.2.8 타임리프 주요 표현식 기호 비교
- 템플릿 내부에서 데이터를 가리키는 기호에 따라 꺼내는 자원(객체, 다국어 메시지 파일, 내장 라이브러리 등)이 완전히 달라지므로 목적에 맞게 구별하여 지정
- 주요 기호 및 용도
  - `${...}` (변수 표현식): 컨트롤러에서 Model에 담아 전달한 일반 자바 객체나 변수의 값을 직접 꺼낼 때 사용
  - `*{...}` (선택 변수 식): `th:object`로 지정해 둔 부모 객체의 내부 필드명을 생략형으로 단축 참조할 때 사용
  - `#{...}` (메시지 표현식): 다국어 지원을 위해 외부 메시지 설정 파일(src/main/resources/messages_언어코드.properties)에 정의된 텍스트를 꺼내올 때 사용
  - `#객체명` (유틸리티 객체 접두사): 중괄호 없이 쓰이며, 날짜 형식 변환이나 검증 필드 에러 검출 등을 처리하는 타임리프 기본 내장 객체를 가리킴 (예: #temporals, #numbers, #fields)

## 6.3 공통 레이아웃 설계 및 컴포넌트 재사용
- 수많은 웹 페이지에서 공통적으로 반복 생성되는 헤더, 네비게이션바, 푸터, 외부 스타일시트(CSS)/자바스크립트(JS) 로드 링크 등을 매번 복사하여 개별 파일에 중복 작성하는 비효율을 방지하고 통합 제어하기 위해 사용함

### 6.3.1 템플릿 fragment 정의와 삽입용 속성의 이해
- `th:fragment`
  - 공통으로 분리하여 재사용할 HTML 마크업 블록(템플릿 fragment)을 정의할 때 사용
  - 태그 내부에 fragment의 고유 식별 명칭을 속성값으로 부여함

  - resources/templates/layout/common.html
  ```html
  <footer th:fragment="footer">
    <p>스프링 게시판</p>
  </footer>
  ```

- `th:replace`
  - 선언된 대상 태그 자체를 지정한 공통 템플릿 fragment 태그로 완전히 대체함
  - 렌더링 결과물에서 원래 감싸고 있던 호출처 태그가 완전히 사라지고 공통 fragment 태그만 남게 됨
  - 일반적으로 헤더 로드나 전체 화면 레이아웃 대체에 가장 보편적으로 활용

  - fragment 사용
    ```html
    <div th:replace="~{layout/common :: footer}">기존 영역</div>
    ```

  - 렌더링 결과
    ```html
    <footer>
      <p>스프링 게시판</p>
    </footer>
    ```

- `th:insert`
  - 선언된 대상 태그의 내부에 자식 요소로 지정한 공통 템플릿 fragment 태그를 끼워 넣음
  - 호출처 태그의 골격을 그대로 보존해야 할 때 사용함

  - fragment 사용
    ```html
    <div th:insert="~{layout/common :: footer}">기존 영역</div>
    ```

  - 렌더링 결과
    ```html
    <div>
      <footer>
        <p>스프링 게시판</p>
      </footer>
    </div>
    ```

- `<th:block>`
  - 타임리프에서 제공하는 유일한 가상 태그로, 화면 렌더링 시 태그 자체는 흔적 없이 사라지고 내부의 콘텐츠만 남음
  - HTML 문서 구조나 스타일 레이아웃을 깨뜨리지 않으면서 th:each나 th:if 같은 제어 속성을 다수의 태그에 동시에 일괄 적용하고자 할 때 사용

  - 사용 예시
    ```html
    <!-- th:block 태그를 사용한 반복 제어 -->
    <th:block th:each="post : ${posts}">
      <div>
        <span th:text="${post.id}">0</span>
        <span th:text="${post.title}">제목</span>
      </div>
      <hr>
    </th:block>
    ```

  - 최종 렌더링 결과(posts에 id가 1, 2인 게시글이 들어있는 경우)
    ```html
    <div>
      <span>1</span>
      <span>1번 게시글</span>
    </div>
    <hr>
    <div>
      <span>2</span>
      <span>2번 게시글</span>
    </div>
    <hr>
    ```

### 6.3.2 템플릿 경로 지정 표현식과 동작 규칙
- fragment를 가져올 때는 물결 괄호 표현식 `~{템플릿파일경로 :: fragment명}`을 기재하여 사용
- 스프링 부트 환경에서 타임리프 템플릿의 기본 탐색 루트 경로는 `resources/templates/`로 자동 사전 설정되어 있음
- 예를들어 `~{layout/common :: nav-header}`로 명시하면, `resources/templates/layout/common.html` 파일 내부에서 `th:fragment="nav-header"`로 지정해 둔 HTML fragment를 가져옴

### 6.3.3 템플릿 fragment에 파라미터 전달하기
- 공통 템플릿 fragment를 호출할 때 함수처럼 파라미터(매개변수)를 전달하여, fragment 내부의 텍스트나 속성을 동적으로 다르게 렌더링할 수 있음
- 정의할 때 `th:fragment="fragment명(매개변수1, 매개변수2, ...)"` 형태로 괄호 안에 변수를 선언하고, 호출할 때 `~{경로 :: fragment명(값1, 값2)}` 형태로 인수를 전달함

```html
<!-- 공통 파일(common.html) 내 파라미터 조각 정의 예시 -->
<th:block th:fragment="nav-header(title)">
  <h1 th:text="${title}">게시글 목록 보기</h1>
  <div class="nav">
    <a th:href="@{posts}">홈으로</a>
    <a th:href="@{posts/new}">글작성</a>
  </div>
</th:block>
```

```html
<!-- 호출처(list.html) 코드 -->
<th:block th:replace="~{layout/common :: nav-header('게시글 목록')}">
  <h1>게시판</h1>
  <div class="nav">
    <a href="list.html">홈으로</a>
    <a href="write.html">새글</a>
  </div>
</th:block>
```

```html
<!-- th:replace와 매개변수 바인딩이 완료된 최종 HTML 구조 -->
<h1>게시글 목록</h1>
<div class="nav">
    <a href="list.html">홈으로</a>
    <a href="write.html">글작성</a>
</div>
```

# 7. 웹 요청 처리와 디스패처 서블릿

## 7.1 HTTP 요청 라이프사이클 및 디스패처 서블릿의 핵심 역할
- 디스패처 서블릿 (DispatcherServlet)
  - HTTP 요청을 가장 먼저 받아 적절한 핸들러(컨트롤러)로 요청을 배분하고, 처리 완료된 결과를 취합하여 클라이언트에게 응답 형태로 반환해 주는 Front Controller Pattern 서블릿
  - 애플리케이션의 모든 웹 요청 관문을 단일 지점으로 통일하여 공통 처리(보안, 다국어 처리 등) 일원화
  - 고급 식당 입구에서 손님을 맞이하고 담당 웨이터(컨트롤러)에게 자리 안내(에스코트)를 수행하는 홀 지배인 또는 안내 직원에 비유 가능

## 7.2 요청에서 응답까지 MVC 구성 요소의 호출 흐름
고급 식당의 운영 구조에 비유하여 디스패처 서블릿을 중심으로 웹 요청이 처리되는 순차적 단계를 설명함

1. 클라이언트 요청 수신 (손님의 입장 및 주문)
  - 손님(웹 브라우저)이 식당에 들어와 원하는 주문 사항을 전달함 (HTTP 요청 전송)
2. 핸들러 조회 (담당 서빙 직원 지정)
  - 홀 지배인(디스패처 서블릿)이 식당의 좌석 및 구역 안내 지도(핸들러 매핑)를 대조하여, 손님이 요청한 주소(URL)를 담당할 홀의 서빙 직원(컨트롤러)이 누구인지 찾아내어 연결함
3. 핸들러 어댑터 결정 (서빙 직원의 응대 방식 조율)
  - 홀 지배인이 해당 서빙 직원(컨트롤러)이 손님의 주문 양식을 원활하게 접수할 수 있도록 돕는 통신 도구 및 규칙(핸들러 어댑터)을 준비시킴
4. 핸들러(컨트롤러) 실행 (서빙 직원의 주문 접수 및 요리 위임)
  - 서빙 직원(컨트롤러)이 호출되어 손님의 주문을 최종 접수하고, 주방의 요리사(모델 및 서비스 계층)에게 요리를 하도록 지시하여 결과 데이터(음식)를 완성함
5. ModelAndView 반환 (서빙 직원의 서빙 대기)
  - 주방에서 완성된 음식 데이터(Model)와 이 요리를 담아낼 알맞은 그릇의 이름(View)을 서빙 직원(컨트롤러)이 모아 하나의 주문 보고서(ModelAndView) 형태로 취합하여 홀 지배인에게 보고함
6. 뷰 리졸버 호출 (그릇 준비실 수색)
  - 홀 지배인이 식기 준비실 담당자(뷰 리졸버)에게 주문 보고서에 적힌 그릇의 이름(예: board/list)을 전달하며, 그에 대응하는 실제 식판이나 접시(물리 뷰 객체)를 찾아 준비하도록 지시함
7. 뷰 렌더링 (음식 플레이팅)
  - 준비실 담당자가 찾아온 식판(뷰) 위에 주방 요리사가 완성한 음식(Model 데이터)을 보기 좋게 올려서 손님이 직접 먹을 수 있는 최종 요리(HTML 화면)로 완성함
8. 응답 반환 (요리 서빙 완료)
  - 홀 지배인이 최종 완성된 요리 화면을 손님(클라이언트)의 식탁 앞에 대접함 (HTTP 응답 전송)

# 8. 데이터 검증 및 예외 처리

## 8.1 Controller 계층에서의 입력값 검증 필요성 및 @Valid 활용
- 클라이언트로부터 전달받은 데이터가 내부 비즈니스 로직이나 저장소에 전달되기 전에 프레젠테이션 계층에서 입력값을 검증 할 필요가 있음
- DTO 필드에 검증 어노테이션(@NotBlank, @Size 등)을 선언하고, 컨트롤러 메서드 매개변수 앞에 @Valid를 부여하여 유효성 검증을 자동화 할 수 있음

- 자주 활용하는 주요 검증 어노테이션
  - @NotNull: 대상 값이 null이 아니어야 함 (모든 타입 대상)
  - @NotEmpty: 대상 값이 null이 아니고, 빈 문자열("")이나 빈 컬렉션(요소 개수가 0개)이 아니어야 함 (문자열, 컬렉션 대상)
  - @NotBlank: 대상 값이 null이 아니고, 공백 문자(" ")를 제외한 최소 1글자 이상의 유효 문자를 포함해야 함 (문자열 전용)
  - @Size(min, max): 문자열의 길이나 컬렉션의 요소 개수가 지정된 최솟값과 최댓값 범위 내에 존재해야 함
  - @Min(value) / @Max(value): 대상 숫자가 지정된 기준값 이상 또는 이하여야 함
  - @Positive / @PositiveOrZero: 대상 숫자가 양수(0보다 큰 수) 또는 0을 포함한 양수여야 함
  - @Email: 대상 문자열이 올바른 이메일 주소 형식(예: user@example.com)에 부합해야 함
  - @Pattern(regexp): 대상 문자열이 지정한 정규 표현식(Regex)의 패턴 규칙과 정확히 일치해야 함

  ```java
  import jakarta.validation.constraints.NotBlank;
  import jakarta.validation.constraints.Size;

  public class PostDto {
      private int id;

      @NotBlank(message = "제목은 필수 입력 항목입니다.")
      @Size(max = 100, message = "제목은 100자 이하로 입력해야 합니다.")
      private String title;

      @NotBlank(message = "내용은 필수 입력 항목입니다.")
      private String content;

      @NotBlank(message = "작성자는 필수 입력 항목입니다.")
      @Size(min = 2, max = 10, message = "작성자 이름은 2자 이상 10자 이하여야 합니다.")
      private String author;
  }
  ```

## 8.2 BindingResult 객체를 이용한 검증 에러 처리
- BindingResult는 데이터 검증 도중 규격 위반 오류가 발견되었을 때, 프로그램이 예외를 발생시키며 중단되는 대신 검증 실패 상세 정보를 수집하여 전달하는 전용 인터페이스
- 컨트롤러 메서드 선언 시 BindingResult 파라미터를 추가해서 사용할 수 있음
- BindingResult 파라미터는 반드시 검증 대상 DTO 객체(예: @Valid가 붙은 객체) 바로 다음에 기술해야 함
- 컨트롤러는 BindingResult.hasErrors() 메서드를 통해 오류 유무를 확인하고, 에러가 존재할 시 비즈니스 처리를 진행하지 않고 본래의 입력 폼 화면으로 돌아가도록 분기 처리

- BindingResult와 타임리프 에러 매핑 동작 메커니즘
  - 스프링 MVC는 DTO 검증 실패 시 생성된 검증 에러 정보를 BindingResult 객체에 저장하여 뷰 템플릿에 모델 속성으로 자동 전달함
  - 타임리프 뷰 내에서는 `th:object="${postForm}"`으로 선언된 객체의 에러 여부를 `#fields.hasErrors('title')` 식으로 검출 가능하며, `th:errors="*{title}"`을 통해 BindingResult에 저장된 해당 필드의 에러 메시지를 태그 텍스트로 동적 출력 처리 할 수 있음

  ```java
  // 게시글 등록 요청을 처리하는 컨트롤러
  @PostMapping("/write")
  public String writePost(@Valid @ModelAttribute("postForm") PostDto post, // Validation 검증 대상 객체
                          BindingResult bindingResult){ // Validation 검증 결과(대상 객체 뒤에 기술해야 함)
      if(bindingResult.hasErrors()){
          return "board/write";
      }
      savePost(post);
      return "redirect:list.html";
  }
  ```

  ```html
  <form th:action="${write}" th:object="${postForm}" method="post">
    <div class="form-group">
      <label>제목</label>
      <input type="text" id="title" th:field="*{title}">
      <div class="field-error" th:if="${#fields.hasErrors('title')}" th:errors="*{title}">에러 메시지</div>
    </div>
    <div class="form-group">
      <label>작성자</label>
      <input type="text" id="author" th:field="*{author}">
      <div class="field-error" th:if="${#fields.hasErrors('author')}" th:errors="*{author}">에러 메시지</div>
    </div>
    <button type="submit">등록</button>
  </form>
  ```

## 8.3 @ControllerAdvice 기반 글로벌 예외 처리
- 전역 예외 처리를 담당하는 @ControllerAdvice 어노테이션 기반 클래스를 설계하여 다수의 컨트롤러에서 예기치 않게 발생하는 예외를 통합해서 처리할 수 있음
- 스프링 MVC는 컨트롤러 외부로 던져진 예외를 가로채서 @ControllerAdvice 빈이 소유한 @ExceptionHandler 메서드 중 예외 타입이 일치하는 메서드를 찾아 자동으로 호출해 줌
- 동작을 위한 필수 요구 조건
  - 스프링이 전역 예외 처리기로 빈 등록을 할 수 있도록 예외 처리기 클래스 상단에 @ControllerAdvice를 부여
  - 처리할 예외별로 메서드를 정의하고 상단에 @ExceptionHandler(예외클래스.class)를 매핑 선언
  - 예외 처리 클래스가 스프링 부트 애플리케이션 컴포넌트 스캔 대상 패키지 하위에 위치해야 함
- 예외 발생 시 에러 원인 메시지를 Model 객체에 담아두고, 에러 페이지 전용 뷰 템플릿(예: templates/error/400.html)을 호출하여 사용자에게 오류 안내 화면을 송출함

```java
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(IllegalArgumentException.class)
    public String handleIllegalArgumentException(IllegalArgumentException ex, Model model) {
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/400"; // templates/error/400.html 뷰 페이지를 렌더링
    }
}
```

# 9. Spring MVC 웹 테스트와 MockMvc

## 9.1 웹 통합 테스트와 MockMvc 개요
- 웹 통합 테스트: WAS 서버를 물리적으로 구동하지 않고도 컨트롤러가 HTTP 요청 주소를 받아 처리하고 뷰 템플릿이 올바르게 반환 및 렌더링 되는지 모의 환경을 빌드하여 일괄 검증하는 기술
- MockMvc: 서블릿 컨테이너를 가동하는 과정 없이 모의 서블릿 환경을 구성하여 컨트롤러의 웹 요청과 응답을 빠르게 시뮬레이션 및 검증하는 테스팅 전용 모킹 엔진

## 9.2 MockMvc 웹 요청 검증 실습
- @SpringBootTest와 @AutoConfigureMockMvc 어노테이션을 결합하여 가상으로 API 및 뷰 컨트롤러 요청을 검증하는 자바 소스 코드를 구성

```java
package net.likelion.bebc25.board.controller;

import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.post;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.model;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.redirectedUrl;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.status;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.view;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.webmvc.test.autoconfigure.AutoConfigureMockMvc;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.web.servlet.MockMvc;

@SpringBootTest
@AutoConfigureMockMvc
public class BoardControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    public void listPostsViewTest() throws Exception {
        mockMvc.perform(get("posts"))
                .andExpect(status().isOk())
                .andExpect(view().name("board/list"))
                .andExpect(model().attributeExists("posts"));
    }

    @Test
    public void createPostValidationFailureTest() throws Exception {
        mockMvc.perform(post("posts/new")
                        .param("title", "")
                        .param("author", "")
                        .param("content", "유효하지 않은 빈 요청입니다."))
                .andExpect(status().isOk())
                .andExpect(view().name("board/form"))
                .andExpect(model().hasErrors());
    }

    @Test
    public void indexRedirectTest() throws Exception {
        mockMvc.perform(get("/"))
                .andExpect(status().is3xxRedirection())
                .andExpect(redirectedUrl("posts"));
    }
}
```

# 10. 스프링 부트 웹 개발 생산성 및 모니터링 도구

## 10.1 Spring Boot Actuator를 활용한 운영 환경 모니터링
- Spring Boot Actuator는 웹 서버가 가동된 후, 현재 구동 중인 애플리케이션의 내부 상태(JVM 메모리 점유율, 쓰레드 개수, 데이터베이스 연결 상태 등)를 HTTP 엔드포인트(/actuator)를 통해 정밀하게 진단하고 노출해 주는 운영 지원 도구
- 모니터링 엔드포인트 종류:
  - `/actuator/health`: 애플리케이션 및 연결된 DB의 작동 여부(Liveness) 모니터링
  - `/actuator/metrics`: CPU 점유율, 메모리 사용량, HTTP 요청 누적 통계 모니터링

- Actuator 의존성 추가 설정:

```groovy
dependencies {
    runtimeOnly 'org.springframework.boot:spring-boot-starter-actuator'
}
```

- 웹 모니터링 엔드포인트 전역 개방 설정:
배포 환경에서는 보안을 위해 노출을 통제하지만, 개발 및 관리망 모니터링을 위해 노출 범위를 설정으로 제어

```properties
# 액추에이터 모니터링 엔드포인트 웹 노출 범위 설정
management.endpoints.web.exposure.include=health,metrics
```
